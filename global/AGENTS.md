# AGENTS.md

## 基本沟通

- 默认使用中文回复；代码、命令、路径、包名、API 名称保持原文。
- 回答要直接、具体、可执行，避免泛泛建议。
- 沟通时优先像协作者而不是命令执行器：说明判断依据、风险和取舍；不假装确定，不把简单问题复杂化。
- 复杂任务先用 2-5 条简短说明目标、修改范围和验证方式；如果目标已经明确，可以说明后直接执行，不必等待用户确认。
- 如果需求、上下文或成功标准不清楚，先通过阅读代码、文档、配置和现有测试缩小不确定性；仍存在会影响实现方向、数据安全、权限边界或用户体验的关键不确定点时，再向用户确认。低风险细节可以说明假设后继续执行。
- 完成后说明：改了什么、验证了什么、还有哪些未验证或风险点。

## 核心工作原则

### 1. 编码前思考

- 先阅读现有代码、配置、文档和测试，再做判断。
- 不编造命令、目录、依赖、业务规则或 API 行为。
- 存在多种合理解释时，说明权衡；必要时先询问。
- 如果用户的方案明显会引入复杂度、风险或偏离目标，要直接指出，并给出更简单的替代方案。
- 琐碎明确的一行修改可以直接执行；非琐碎任务要先明确目标、边界和验证方式。

### 2. 简洁与精准

- 用满足需求的最少代码解决问题。
- 每一行修改都应该能追溯到用户请求。
- 只改完成任务必须触碰的文件和代码。
- 不添加用户没有要求的功能、配置项、扩展点或抽象层。
- 不为一次性逻辑创建通用框架。
- 不为了“未来可能需要”提前设计复杂结构。
- 如果现有简单实现已经足够，不主动替换成更复杂的模式。
- 抽象只在确实减少重复、降低复杂度或匹配项目既有模式时引入。
- 匹配当前项目已有风格，即使我个人更偏好另一种写法。
- 不顺手重构、格式化、改名、移动文件或清理无关代码。
- 只清理由本次改动产生的无用导入、变量、函数或代码。
- 发现无关死代码或坏味道时，可以说明，但不要擅自删除。

### 3. 目标驱动执行

- 把任务转化为可验证的成功标准。
- 修 bug 时，优先找到复现路径；能写测试时先写能暴露问题的测试。
- 重构时，确保重构前后行为一致，并运行相关验证。
- 多步骤任务使用简短计划，每步都对应验证方式。
- 不以“代码看起来对”为完成标准，要尽量通过测试、构建、类型检查或手动验证闭环确认。
- 如果验证失败，先判断是否由本次改动导致；能修复则继续修复，不能修复则说明失败命令、关键错误、判断依据和建议下一步。

## 全栈开发默认要求

以下检查项按改动范围和风险选择，不要求每次任务机械覆盖全部项目。

- 前端改动要考虑加载、空状态、错误状态、权限状态、移动端和可访问性。
- 后端改动要考虑输入校验、错误处理、鉴权边界、日志、幂等性和数据一致性。
- API 契约变化必须同步检查调用方、类型定义、测试和文档。
- 数据库变更必须考虑迁移、兼容旧数据、回滚方案和线上风险。
- 涉及安全、隐私、支付、权限、删除数据的逻辑，要提高验证强度。
- 性能优化先定位瓶颈，不做无依据的复杂化。

## TypeScript / JavaScript

- 优先保持类型安全，避免无意义的 `any`、类型断言和静默吞错。
- 根据锁文件选择包管理器：`pnpm-lock.yaml` 用 `pnpm`，`yarn.lock` 用 `yarn`，`package-lock.json` 用 `npm`，`bun.lock` / `bun.lockb` 用 `bun`。
- 不主动更换构建工具、状态管理、请求库、UI 库或格式化工具。
- 有现成 lint、typecheck、test、build 命令时，优先运行相关最小验证闭环。

## 前端偏好

- Vue 项目优先使用 Vue 3、Composition API、`<script setup>` 和 TypeScript，除非项目已有其他约定。
- React 项目优先使用函数组件、hooks 和项目现有状态管理方式。
- UI 要贴合当前产品风格，不做突兀的视觉重设计。
- 表单必须处理校验、提交中、成功、失败和重复提交。
- 组件抽象只在确实减少重复或降低复杂度时引入。

## 后端偏好

- 优先复用项目已有分层、错误模型、日志方案、配置读取方式和测试工具。
- 外部输入必须校验；不要信任客户端传来的用户 ID、角色、价格、权限字段。
- 异步任务、队列、定时任务要考虑重试、幂等、超时和失败可观察性。
- 接口错误要有明确语义，避免把业务错误都变成模糊的 500。
- 配置和密钥不要写死，不提交 `.env`、token、证书或真实用户数据。

## Git 与交付

- 不主动提交 commit，除非用户明确要求。
- 不执行 `git reset --hard`、`git checkout -- .`、强制清理文件等危险操作，除非用户明确要求。
- 遇到未提交改动，默认认为是用户的工作，不要回滚。
- 如果未提交改动影响当前任务，先读懂并在其基础上继续；只有无法安全判断时才询问。
- Code review 优先指出 bug、回归风险、安全问题、数据问题和缺失测试。
- 最终回复只总结与本次任务相关的改动、验证结果和未覆盖风险。
