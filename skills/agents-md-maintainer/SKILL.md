# AGENTS.md Maintainer

## 输出语言

默认使用中文输出，包括最终回复、审查结论、新建或整理后的 `AGENTS.md`、拆分到 `docs/` 的文档内容。

只有在用户明确要求英文，或项目现有文档明显全英文且用户要求保持一致时，才改用英文。

保留命令、路径、包名、API 名称、代码标识符和工具名称的原文。

## 第一原则：渐进式披露

`AGENTS.md` 的第一原则是渐进式披露：它是一张项目地图，不是一本开发手册。

只有两类内容应该直接写进根目录 `AGENTS.md`：

- AI 理解项目全貌的必要信息：技术栈、仓库结构、核心模块、分层架构。
- 违反会直接导致问题的硬性规则：编码规约、命名约定、依赖边界、禁止项、必须执行的验证步骤。

判断标准：如果 AI 不知道这条信息就会写出错误代码，放进 `AGENTS.md`；如果只是写出不够好的代码，放进详细文档，`AGENTS.md` 里只保留摘要和链接。

因此，教程、长示例、实现细节、模块深潜、背景知识和“只是让代码更好”的建议，应移出根 `AGENTS.md`。根文件可以保留从仓库真实文件中发现的关键命令、最小验证闭环，以及指向详细文档、源码、schema、类型定义或参考项目的短链接。

本 skill 应能作为独立目录导入其他项目使用，不依赖本仓库的 `example/` 目录。`example/` 只是维护本 skill 时可参考的示例产物。

## 工作流程

1. 编辑前先检查仓库：文件列表、manifest、构建文件、包管理器文件、CI workflow、现有文档，以及已有 `AGENTS.md`。
2. 从权威来源发现命令：`package.json`、`Makefile`、`pnpm-workspace.yaml`、`turbo.json`、`pyproject.toml`、`Cargo.toml`、`go.mod`、`pom.xml`、Gradle 文件、Docker Compose、脚本和 CI workflow。不要编造命令。
3. 如果缺少 `AGENTS.md`，在仓库根目录创建；如果已经存在，保留正确的项目专属规则，只提升清晰度、真实性和可维护性。
4. 如果现有指令过长或像手册，优先拆到已有文档；没有合适位置时，在 `docs/` 下创建最少必要文档。
5. 对 monorepo 或不同子系统，只有子目录存在特殊规则时才新增嵌套 `AGENTS.md`。嵌套文件应短小，只描述当前目录规则。
6. 根据 bad case 改规则时，先明确 AI 犯了什么错，再写能避免复发的最小规则，最后选择落点：根规则、模块文档、组件模式文档、参考导航文档或自动化检查。
7. 编辑后运行安全且相关的验证命令。无法验证时，在最终回复中说明原因。

## 文件边界

不要为了套模板创建空目录或空文档。只创建或更新当前仓库真正需要的文件。

| 路径 | 用途 | 根 `AGENTS.md` 保留内容 |
| --- | --- | --- |
| `AGENTS.md` | AI Coding 入口 | 心智模型、真实命令、硬规则、验证闭环、文档导航 |
| `README.md` | 人类项目说明 | 必要时交叉链接；不要复制完整 AI 规则 |
| `docs/architecture-server.md` | 后端分层、依赖规则、领域模型 | 后端架构摘要和链接 |
| `docs/architecture-web.md` | 前端分层、依赖规则、状态和 API 边界 | 前端架构摘要和链接 |
| `docs/development.md` | 环境、安装、启动、常用命令 | 命令摘要和链接 |
| `docs/validation.md` | 测试、curl、浏览器检查、日志、Token | 最小验证流程和链接 |
| `docs/conventions.md` | 命名、编码约定、通用规则 | 关键硬规则和链接 |
| `docs/server*.md` / `docs/web*.md` | 子系统细节 | 子系统摘要和链接 |
| `docs/design-docs/*` / `docs/patterns/*` | 组件模式、ADR、模块深潜、参考导航 | 一句话导航 |
| `reference-projects/*` | 只读参考源码 | 参考优先级和导航链接 |

优先复用项目已有文档位置。只有内容天然会形成系列时，才新增一层目录，例如 `ref-*.md`、`*-patterns.md`、ADR 或模块深潜。

## 根文件建议结构

结构应服务于仓库实际情况。小项目可以更短；已有更清晰约定时沿用项目约定。

```markdown
# AGENTS.md

## 1. 项目概述

一段话说清楚：项目是什么、技术栈、仓库结构。

前 10 行必须让 AI 建立项目心智模型。

## 2. 快速命令

构建、启动、格式化、质量检查的命令速查表。

环境变量配置说明（env 文件位置、启动脚本自动 source）。

## 3. 后端架构

包结构树（ASCII）+ 每个包的用途注释。

核心子系统的简要说明 + 详细文档链接。

前后端术语映射（如有差异）。

## 4. 前端架构

技术栈、路由方案、API 层约定、组件库规范。

详细文档链接。

## 5. 关键约定

5-10 条硬性编码规则（违反会直接导致问题的）。

每条规则附详细文档链接。

## 6. 本地开发及验证流程

「改 → 构建 → 启动 → 验证」的完整闭环。

curl 验证模板、Token 获取、日志路径。

## 7. 质量检查

lint、format、build、test 命令矩阵。

## 8. 参考项目约定

参考项目列表 + 优先级规则。

## 9. 文档导航

所有详细文档的索引表。
```

根 `AGENTS.md` 建议控制在 200 行以内；超过后拆分。

## 写作要求

- 写给已经在仓库里的 AI Agent，直接、明确、可执行。
- 优先使用源码、schema、类型定义、脚本和 CI 中的事实。
- 不写“保持代码整洁”这类泛泛建议，除非项目有明确标准和检查方式。
- 不确定的信息要标为假设，或先不写入规则。
- 示例必须短且贴合当前仓库；长示例拆到文档。
- 可以记录敏感配置的位置和优先级，但不要建议提交密钥或本地私有配置。

## 拆分规则

以下内容应拆出根 `AGENTS.md`：

- 分步骤教程。
- 长示例、API 细节、截图说明或 curl 集合。
- 只适用于单个模块、包、组件族或参考项目的内容。
- 有助于提高质量但不属于“避免写错”的背景知识。
- 让根文件变得难以扫描的内容。

默认拆分位置：

- 后端架构和模块关系：`docs/architecture-server.md`
- 前端架构和模块关系：`docs/architecture-web.md`
- 环境和命令：`docs/development.md`
- 验证、日志、Token、浏览器或 curl 检查：`docs/validation.md`
- 编码约定和命名规则：`docs/conventions.md`
- 后端或前端细节：`docs/server.md`、`docs/web.md` 或对应子目录
- 组件模式和长示例：`docs/patterns/` 或 `docs/design-docs/`
- 参考项目、私有框架、相邻系统：`docs/design-docs/ref-*.md` 或 `docs/references/*.md`

根 `AGENTS.md` 对每个拆分主题只保留一句摘要和链接。

## 检查清单

结束前确认：

- 前几行能快速说明项目用途、技术栈和重要目录。
- 命令都来自仓库真实文件。
- 根文件只保留硬规则和高价值导航。
- 验证流程具体，不只是“运行测试”。
- 详细内容通过链接渐进披露，而不是复制进根文件。
- 最终回复说明改了什么、运行了什么验证、哪些信息仍是假设。

## 最终回复

如果完成了编辑，简要说明修改范围、规则如何合并或拆分、验证命令是否运行。

如果只是审查，先列最重要的问题，再给出精简建议。
